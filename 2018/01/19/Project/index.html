BMP格式解析

BMP格式在维基百科里解释的很详细。下面这张来自维基百科的图片很好的展示了BMP格式的结构，以及各个field代表的意义。

图中被我用红色框圈起来的部分是我们在这次实验真正关心的数据。第一个框内的数据表示PixelArray在文件中的偏移量，PixelArray是这个图片真正的RGB数据，我们最终就是使用这部分的数据在LCD上显示图片。PixelArray 的相关内容在后面详细说明。

第二个框里的数据分别为图片的宽和长。要显示图片，这两个数据显然是很重要的。

第三个框就是刚才提到的PixelArray，里面存储了图片每个Pixel的RGB值，所以这个部分是最重要的。

PixelAraay说明

The pixel array is a block of 32-bit DWORDs, that describes the image pixel by pixel. Usually pixels are stored “upside-down” with respect to normal image raster scan order, starting in the lower left corner, going from left to right, and then row by row from the bottom to the top of the image.

从维基百科的这段话可以知道，在PixelArray中存储的图片是上下颠倒的。即这个部分最开始的数据是图片最左下角的数据。因此我们在LCD上生成图片的时候要注意，从左下角开始生成，否则图片会上下颠倒。

当我们采用24位BMP时，每个Pixel由24位数字决定颜色，RGB三个值，每个值各占8位，范围在0~255。要注意的是这3个8位数据的顺序不是我们常说的R, G, B，而是B, G, R。在解析的时候要注意这个顺序，否则生成的图片会很奇怪。

从上面解释BMP格式的图片中可以看到，在PixelArray中每一行的末尾都有一些Padding，这个BMP规定了每一行数据长度都要是4个字节的倍数，所以不同的图片padding所占用的字节数不尽相同。百科里给出了每一个数据长度的公式：

​ $$RowSize = \lfloor {BitsPerPixel \times ImageWidth + 31 \over32} \rfloor \times 4$$

我们在做解析的时候要注意跳过Padding部分的数据。

实现图片放大功能

这是一个简单的数字图像处理。用逆向差值就可以实现了。如果f(x, y)为原图像，g(x, y)为放大后的图像。那么如果把宽变为为原来的n倍，高变为为原来的m倍，那么两个图像有这样的关系:

$$f(x, y) = g(n x, m y)$$

如果原图像的宽和高为width和height，那么放大后的就是n * width和m * height。显然，我们是用两层循环来将图片写入LCD上的，但是循环遍历的是哪个图片很重要。如果循环遍历的是原图像，那么就是将原图像的(x, y)像素填入放大图像的(n * x, m * y)像素，这样的话在LCD上显示出来的图片也只有width * height个，但实际上应该要有m * n * width * height，这就造成了像素之间有空白点出现。

为了这个问题，我们可以反过来想，不遍历原图像，而是遍历放大后的图像，于是使用另一个公式:

$$g(x, y) = f({x \over n},{y \over m})$$

也就是说，如果要填充放大后图像的(x, y)坐标，我们就用上述公式找到它在原图像上的像素(x / n, y / m)。这里我们直接利用了整型除法的截断特性，保证每次得到的结果都是整数。当然这种粗暴的做法不是很好，比较理想的做法应该是计算得到x/n和y/n的真实值，即不截断，然后根据结果找到离这个结果最近的四个像素点，然后通过离这四个点的距离进行加权平均得到像素值再填入放大图像的(x, y)上。我们这次project没有采用这种方案，而是直接截断，原因很简单，因为方便实现。下面是这个函数的代码：

中断处理函数

因为要实现相片的循环播放，所以我们需要用一个变量来记录当前显示的是哪一图片。我们通过一个静态变量static int j来完成这件事。将它声明为static或是全局变量都可以，只要每次执行do_irq的时候，它都能够正确的表示当前显示的是哪一张图片就好。将它声明为int是因为所有图片都已经存储在了images数组当中，所以对图片的访问就是对这个数组的访问，这个数组存储了所有图片的起始地址，对这个变量进行数组访问就可以得到对应的图片，所以将j声明为int就可以代表当前显示的是哪一张图片。而要获得下一张图片只要做j = (j + 1) % IMAGENUM计算就可以了。因为是循环播放，所以在加一之后还要做取模运算，注意IMAGENUM是在images.h头文件当中用于记录图片数量的宏定义。

经验总结

消除按键抖动

在先前的按键中断时间中已经遇到过按键抖动问题，在这个project实现当中我们同样遇到了这样的问题。

问题描述

在实现了按键中断功能之后会发现有时候按下一次按键，对应的功能会生效多次。如果是设置为电平触发的话，那产生这个现象很正常。但是即使是设置成边沿触发也同样会有相同的现象产生。

问题原因

这张图片是按下按键之后，对应按键产生的电平变化。理想状况下，一次按键按下与松开，上升沿和下降沿应该各出现一次。但是实际上，从图中可以明显看出按下一次按键，下降沿出现了不止一次。这种现象就是按键抖动。

问题解决

从图中可以看出来，每次按下按键，首先一定会出现一次下降沿。这个下降沿是我们真正需要的。也就是说，按下按键之后第一次执行的do_irq函数是我们真正需要的。如果不做其它处理，那么按下一次按键，这个函数会执行数次。所以我们需要有能力在第一次函数执行的时候避开后面的中断触发。

首先要认识到一点，当按键中断发生的时候，其它按键中断会被暂时屏蔽掉，也就是说，程序在do_irq中运行的时候后面即使有其它按键中断产生，也不会打断当前中断代码的执行。所以在第一次中断产生的时候，我们可以通过delay函数使程序避开图中的前沿抖动阶段，直到进入了稳定的电平状态才开始执行并退出中断处理函数，然后清空中断悬挂寄存器。这样就可以做到按下一次只有一个下降沿触发产生。这就是do_irq函数中delay以及后面的电平判断的作用。

问题原因

这个问题产生的原因是ARM在硬件上不支持除法操作，需要用软件模拟。图中的四个函数就是用来模拟除法的。链接器在链接的时候不知道这些函数的位置，所以产生了上面的错误。raise还是貌似是用来处理除以0的情况。

问题解决

因为除法是很基本的操作，所以通常编译器都会提供相应的软件模拟代码。而我们使用的交叉编译器也同样提供了相应代码，只是链接器并不知道，需要我们指定。

在链接器命令后面用-L指定静态链接库路径，这个库在编译器的相关目录下。我们需要的库是libgcc.a，所以用-lgcc来链接这个库。上面提到的四个函数就实现在这个库当中。链接了这个库之后，再次编译，有可能出现第二幅图的报错，提示说缺少raise函数。这个函数我们自己写就好，不需要执行什么内容：

window
